# "오늘 나갈까?" Lean MVP 전략 — 아이디어 브리프 (수정판)

> 아이디어 브리프 | 2026-04-08
> 기존 브리프의 근본적 약점을 인정하고 전략을 수정한 버전

---

## Executive Summary

**솔직한 현실 인식**: 공공 API 데이터를 예쁘게 보여주는 것만으로는 매일 앱을 열 이유가 없다. 미세미세/네이버 날씨로 충분하기 때문이다. AI를 얹어도, 커뮤니티를 붙여도, "아침 외출 체크" 자체가 매일 앱을 열 만큼의 Pain이 아니다.

**전략 전환**: 과도한 기획을 멈추고, **빠르게 만들어서 내놓고 → 실제 사용자 반응에서 "열 이유"를 찾는다.**

---

## 1. 왜 기존 설계가 약한가

### 냉정한 자가 진단

| 기존 설계 요소 | 왜 매일 열 이유가 안 되는가 |
|---------------|--------------------------|
| 공공 API 데이터 통합 | 미세미세/네이버 날씨에도 있음. 데이터가 새롭지 않음 |
| "3초 스캔" UX | 편리하지만, 미세미세 위젯으로도 1초에 가능 |
| 옷차림 추천 | 처음엔 신기하지만 2주 뒤엔 감으로 입힘 |
| 감염병 알림 | 시즌에만 필요. 연중 매일 볼 이유 안 됨 |
| AI 예측 | 기상청 예보와 크게 다르지 않을 수 있음 |
| 동네 부모 집단 판단 | 처음만 신기하고 끝 |
| 게이미피케이션 | 억지스러움 |

### 핵심 문제

> **"아침에 아이 데리고 나갈까?" 는 부모가 적응해서 사는 수준의 불편이지, 앱을 매일 열 만큼의 Pain이 아니다.**

이 문제를 해결하지 않으면 어떤 기능을 얹어도 DAU가 유지되지 않는다.

## 2. 새로운 전략: Lean MVP → 사용자 반응 기반 진화

### Phase 1: 일단 빠르게 만들어서 내놓는다 (1~3일)

```
목표: "이게 진짜 필요한지" 검증
방법: API 매쉬업 최소 버전 출시 → 사용자 반응 관찰

만들 것:
  - 날씨 + 미세먼지 + 자외선 = 한 화면 통합 (공공 API 3개)
  - 감염병 유행 현황 카드 (질병관리청 API)
  - 기온별 아이 옷차림 추천 (Rule-Based)
  - 외출 종합 점수 (가중치 공식)
  - benefit.dodam.life 서브도메인 배포

만들지 않을 것:
  - AI 예측 엔진 (아직 필요 검증 안 됨)
  - 동네 부모 커뮤니티 (사용자 없이 커뮤니티 = 유령 도시)
  - 게이미피케이션 (억지스러움)
  - 사용자 계정/로그인 (프릭션 최소화)
```

### Phase 2: 사용자 반응 관찰 (출시 후 2~4주)

```
측정할 것:
  1. 재방문율 (D1, D7, D30 리텐션)
     → "매일 오는가?" vs "한번 보고 끝인가?"

  2. 시간대별 사용 패턴
     → 정말 아침 7~8시에 몰리는가?
     → 아니면 미세먼지 나쁜 날에만 오는가?

  3. 어떤 카드를 가장 오래 보는가?
     → 미세먼지? 옷차림? 감염병?
     → 의외의 사용 패턴이 나올 수 있음

  4. 카카오톡 공유율
     → 어떤 상황에서 공유하는가?
     → 바이럴이 자연 발생하는가?

  5. 직접 피드백 수집
     → "이 앱에 뭐가 더 있으면 매일 올 것 같아요?" 원라인 서베이
```

### Phase 3: 데이터 기반 피벗/강화 (4주 후)

```
시나리오 A: "매일 오는 사람이 있다"
  → 왜 오는지 파악 → 그 이유를 극대화
  → AI 예측, 개인화 등 기존 아이디어 접목

시나리오 B: "미세먼지 나쁜 날에만 온다"
  → 시즌 서비스로 피벗
  → 미세먼지+감염병 시즌에 푸시 알림 중심으로 전환
  → "매일 열게 하려는" 욕심을 버리고, "필요할 때 바로 확인" 포지셔닝

시나리오 C: "감염병 정보에 반응이 강하다"
  → 감염병 알림 서비스로 피벗
  → "우리 동네 수족구 급증!" 같은 실시간 알림이 핵심
  → 날씨/미세먼지는 부가 정보로 격하

시나리오 D: "아무도 안 온다"
  → 이 아이디어 자체를 버리고 Day 2 서비스로 이동
  → (혜택 스캐너, 소아과 119 등 다른 매쉬업 서비스)
```

## 3. MVP 기술 스펙 (최소 구현)

### API 구성 (전부 무료)

| API | 용도 | 갱신 주기 | 호출 제한 |
|-----|------|----------|----------|
| 에어코리아 대기오염정보 | PM2.5, PM10 | 30분 | 500건/일 |
| 기상청 단기예보 | 기온, 강수, 바람 | 1시간 | 10,000건/일 |
| 기상청 생활기상지수 | 자외선 | 6시간 | 10,000건/일 |
| 질병관리청 감염병 발생현황 | 감염병 통계 | 1주 | 1,000건/일 |

- Next.js (도담 기존 스택)

- PWA (홈 화면 추가 가능)

### 백엔드

- Next.js API Routes
- 공공 API 캐싱 (Redis 또는 인메모리)
- 분석: 간단한 이벤트 로깅 (재방문율, 카드 클릭 등)

### 배포

- today.dodam.life 서브도메인
- Vercel (기존 인프라)

## 4. AI는 언제 도입하는가

### 지금 아닌 이유

1. **검증 전 최적화는 낭비** — 아무도 안 오는 앱에 AI를 얹어봤자 의미 없음
2. **학습할 데이터가 없음** — 사용자 피드백이 쌓여야 AI가 학습 가능
3. **비용 리스크** — LLM 비용이 DAU 대비 수지 안 맞을 수 있음

### 도입 조건 (Phase 3+)

```
IF D7 리텐션 > 30%:
  → "매일 오는 사용자가 있다" 확인됨
  → AI 예측 (시간대별 점수 변화) 도입 의미 있음
  → LLM 한줄 요약 실험 (A/B 테스트)

IF 피드백에서 "예측" 요구 반복:
  → "오후에 어떻게 되나요?" 질문이 많으면
  → 시간대별 예측 기능 추가

IF MAU 1만+ 도달:
  → 충분한 행동 데이터 축적됨
  → ML 모델 학습 시작 (개인화, 지역 패턴)
  → 데이터 해자 구축 본격화
```

### AI 아이디어 브리프 유지

기존 `docs/ideas/ai-outing-predictor-2026-04-08.md`는 그대로 유지.
Phase 3에서 AI 도입 시 해당 브리프를 참조하여 구현.

## 5. "열 이유" 후보 — Phase 3에서 검증할 것

사용자 데이터가 쌓인 후 A/B 테스트로 검증할 아이디어들:

| 후보 | 가설 | 검증 방법 |
|------|------|----------|
| AI 한줄 요약 | "등원 때 나가세요! 오후엔 주의" → 한줄이 점수보다 재방문 유도 | A/B: 점수만 vs 점수+한줄 |
| 푸시 알림 | "오늘 미세먼지 나쁨 예상" 아침 7시 푸시 → 앱 열게 만듦 | 푸시 ON/OFF 그룹 비교 |
| 감염병 긴급 알림 | "우리 동네 수족구 급증!" → 이게 진짜 킬러일 수 있음 | 감염병 시즌에 리텐션 변화 측정 |
| 어제의 판단 결과 | "어제 GO 73%, 오늘은?" → 매일 다른 숫자가 호기심 자극 | 피처 추가 전후 리텐션 비교 |
| 위젯 | 앱 안 열어도 홈화면에서 점수 확인 → 위젯이 리마인더 역할 | 위젯 설치자 vs 미설치자 리텐션 |

## 7. 이 전략이 기존 브리프와 다른 점

| 항목 | 기존 브리프 | Lean MVP 전략 |
|------|-----------|--------------|
| AI 엔진 | Day 1부터 하이브리드 ML+LLM 설계 | 검증 후 도입 (Phase 3+) |
| 커뮤니티 | 동네 부모 집단 판단 공유 | 사용자 없이 커뮤니티 = 유령도시. 보류 |
| 게이미피케이션 | 스트릭/배지/레벨 | 불필요. 삭제 |
| 옷차림 AI | 개인화 학습 | Rule-Based로 충분. AI는 나중에 |
| 구현 기간 | 7~8일 | 2~3일 (최소한만) |
| 성공 기준 | DAU = 매일 여는 습관 | **D7 리텐션 > 30%** (이것부터 확인) |
| 실패 시 | 고려 안 됨 | Day 2 서비스로 이동 (혜택 스캐너 등) |
